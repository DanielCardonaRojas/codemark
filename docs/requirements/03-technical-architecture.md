# Codemark: Technical Architecture

```
┌──────────────────────────────────────────────────────────┐
│                      CLI Layer (clap)                     │

└──────────────┬───────────────────────────────┬───────────┘
               │                               │
       ┌───────▼────────┐             ┌────────▼────────┐
       │  Bookmark       │             │  Output         │
       │  Engine         │             │  Formatter      │
       │                 │             │  (table/json)   │
       └───┬────┬────┬──┘             └─────────────────┘
           │    │    │
    ┌──────▼┐ ┌▼────▼──────┐
    │ Query │ │ Resolution  │
    │ Gen   │ │ Engine      │
    └───┬───┘ └──┬─────┬───┘
        │        │     │
   ┌────▼────────▼┐  ┌▼──────────┐
   │  Tree-sitter  │  │  Git       │
   │  Parser       │  │  Context   │

   └───────────────┘  └───────────┘

   │         Storage (rusqlite)     │
   │         .codemark/codemark.db  │
   └────────────────────────────────┘
```

## Language Support

Initial language support (Phase 1):
- **Swift** — primary test language given the user's context
- **TypeScript / TSX**
- **Rust**
- **Python**

Each language requires:
1. A compiled tree-sitter grammar (linked at build time or loaded dynamically).
2. A query generation strategy that understands the language's named node types.
3. Test fixtures exercising bookmark creation and resolution.

Language grammars are compiled into the binary using tree-sitter's build toolchain. This keeps the binary self-contained.

## Storage

### Location
The SQLite database lives at `.codemark/codemark.db` relative to the **git repository root**. Codemark detects the repo root via `git rev-parse --show-toplevel` or `git2`.

### Schema Versioning
Migrations are embedded in the binary and run on first access. The database stores a `schema_version` in a metadata table. Each migration is idempotent.

### Concurrency
SQLite WAL mode is enabled for concurrent reads. Write operations acquire an exclusive lock via SQLite's built-in locking. For the expected usage pattern (single CLI invocation at a time), this is sufficient.

## Error Handling Strategy

- **User-facing errors**: Displayed with context and suggestions. No stack traces.
- **Resolution failures**: Never crash. Record the failure, update status, continue with remaining bookmarks.
- **Missing grammars**: Clear error message naming the language and how to check supported languages.
- **Git errors**: Degrade gracefully — bookmarks work without git context, `commit_hash` is nullable.
- **Database errors**: Migration failures are fatal with clear messaging. Query failures are recoverable.

All errors use a unified error type via `thiserror` that maps to appropriate exit codes.

## Performance Considerations

1. **Parser caching**: Within a single CLI invocation, each file is parsed at most once. Parsed trees are held in a `HashMap<PathBuf, Tree>`.
2. **Batch file grouping**: When resolving multiple bookmarks, group by file path to minimize parsing.
3. **Lazy grammar loading**: Only load grammars for languages actually needed by the current operation.
4. **SQLite pragmas**: Enable WAL mode, set `synchronous = NORMAL`, use prepared statements.
5. **Query compilation**: Tree-sitter queries are compiled once and reused across matches within the same file.
